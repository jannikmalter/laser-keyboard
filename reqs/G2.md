# G2 — Standalone Raspberry Pi / ArtNet build

**Goal.** Make the laser keyboard a self-contained appliance that needs no PC running
QLC+. A Raspberry Pi 3B+ (PoE) lives inside the keyboard enclosure; the keyboard
connects to the Pi over USB, and the whole unit reaches the rest of the installation
over a single network cable (PoE = power + data). The Python program generates
**ArtNet** (DMX-over-UDP) directly, replacing QLC+ + loopMIDI + the DMX-USB interface.

This is a **new program** (`standalone/laserkbd/`), separate from the QLC+ build's
`laserkeyboard.py` (which stays as-is for reference / fallback). The new build also
folds in the QLC+ build's bug fixes (B1–B6) so they aren't reintroduced.

## Architecture

- **MIDI thread** — open the keyboard, read note on/off, maintain key state.
- **DMX thread** — runs on a **free-running, configurable tick**. Lighting calculation
  and ArtNet send happen together on the same tick (in sync), so every computed frame
  is the frame that goes out — no drift between animation and output. Uses a monotonic
  deadline loop so timing doesn't drift; aims for reasonably steady ticks rather than
  hard real-time (see jitter budget). (Outputting a full frame every tick also makes
  the QLC+ "redundant-message" bug, B3, moot by design.)
- **Tick rate may exceed the ~44 Hz DMX ceiling.** The target ArtNet node can be
  configured to send fewer DMX channels per universe, which raises the achievable DMX
  refresh rate above the standard 512-channel limit. The Python tick rate is therefore
  a free choice, bounded by the node's channel count and the Pi's CPU, not by a
  hardcoded 44 Hz. **Target: ~100 Hz.**
- **Jitter budget is generous — don't over-engineer.** At ~100 Hz (10 ms period), even
  ±50% jitter (±5 ms) is imperceptible for this installation, so heavy real-time
  measures (PREEMPT_RT kernel, `isolcpus`, core pinning) are **out of scope unless
  problems actually appear**. "Reasonable" effort only: a monotonic deadline-based
  loop, keep the DMX tick lean, and avoid obvious GIL stalls.
- **Web thread (Flask)** — small interface to edit settings and view logs.
- Thread-safe shared state (key state + settings) with a lock or queue; config
  persisted to disk so settings survive a restart.

## Resilience (R29–R31)

The appliance runs unattended inside the keyboard enclosure, so it must tolerate the
three field failures without manual intervention:

- **USB MIDI disconnect (R29).** *Implemented in `midi_thread.py` (`_port_gone` /
  `_disconnect`); validated on the Pi by a real unplug (2026-06-24).* The MIDI thread must not die
  when the keyboard is unplugged or browns out: reads/opens are wrapped so a disconnect
  surfaces as a caught error, held key state is released (no beam stuck on), and the
  thread retries opening the device by name on a poll loop until it returns. Real
  disconnect detection does **not** trust `is_port_open()` (which can keep reporting
  open after an unplug); it checks the connected port name is still in `get_ports()`,
  and treats any probe exception as "gone". Mirrors the QLC+ build's reconnect intent
  (R3) but with real disconnect detection (B6 in that build is the cautionary case).
- **Network interruption (R30).** ArtNet is UDP, so a downed network/node shows up as a
  send error (or silently dropped packets). The render loop catches send exceptions and
  keeps ticking on its monotonic deadline — it must not block or crash — and output
  simply resumes when the link returns. No buffering of stale frames; the next tick
  sends the current frame. *Validated 2026-06-24 by pulling the PoE cable.*
- **Power loss (R31).** No graceful shutdown is guaranteed, so: systemd auto-starts the
  service on boot (R27) and config writes are atomic (write temp + `os.replace`) so a
  cut mid-write can't corrupt the settings file (R26). Filesystem-level hardening
  (e.g. read-only root / overlay) is optional and out of scope unless corruption shows
  up in practice. *Validated 2026-06-24: because the Pi is PoE-powered, pulling the one
  cable cuts power and network at once — the unit reboots unattended and resumes clean.*

## Milestone 2 — effects

The first slice (R38–R41) brings the QLC+ build's chord handling (R8–R11) into the
standalone build and adds a small effects engine with the first two effects. **This slice
is complete and hardware-confirmed on the Pi + BeamBars (2026-07-22)** — R38–R41 all ☑.
Later slices add the full-keyboard bonus and broader effect parity.

### Chord detection (R38)

The standalone counterpart to QLC+ R8/R9, but generalised from fixed key-index sets to
chord **quality**. The QLC+ build matches two hardcoded triples (`ACCS`) with a NumPy
`accords - keys*accords` trick; here `chords.quality()` (plain Python, evaluated on each
DMX tick) reduces the held keys to their distinct pitch classes (mod 12) and asks whether
they form a **major** triad ({root, +4, +7}) or a **minor** triad ({root, +3, +7}). So
*every* major chord (any root, any inversion/voicing, octave doublings folded in) reads as
"major" and every minor chord as "minor" — the request that drove the change.

- **A plain triad only:** exactly three distinct pitch classes. A fourth distinct pitch
  class (e.g. a seventh) cancels detection, and augmented/diminished/suspended triads match
  neither quality. Pitch-class reduction makes it inversion- and octave-invariant for free.
- **Distinguishing major from minor.** Both have the unordered gap multiset {3, 4, 5}
  around the octave, so the difference is the *order*: read upward from the root a major
  triad is 4 then 3, a minor triad 3 then 4. `quality()` tests each of the three rotations
  (any pitch class can be the root) for the ordered pair (4, 3) → major or (3, 4) → minor.
- **Edge-detected**, not level: the held quality maps (via `chord_effects`) to at most one
  effect, which fires once when the qualifying chord completes and clears once it breaks —
  the DMX thread holds the active effect's trigger time between ticks (`_active_effects`,
  keyed by effect name; the one piece of effect state, per-beam render stays stateless).
  This is also where the QLC+ "redundant-message" flood (B3) is avoided by construction.
- A chord's keys are still per-key beams too; detection reads the same `KeyState`
  snapshot the renderer already takes, so no extra MIDI-side work.

### Effects engine (R39)

A recognised chord activates a **named effect**. Effects are full-array animations:
they address all **40 beams** (4 bars × 10), not just the 32 playable keys, so they
need a whole-array channel enumeration. Today `fixtures.beam_channel()` and
`active_bar_bases()` are scoped to `key_count` (32); the engine needs a sibling that
walks every bar in `bar_base_addresses` × `beams_per_bar` and drives **all four bars'**
channel 1 to `DMX_MODE_PER_BEAM` while an effect is active (otherwise the unmapped bar
stays dark).

- **Stateless render, like `decay.py`.** Each effect is a pure function of *elapsed*
  time since the chord triggered (`now - trigger_time`, monotonic) → a 40-element
  brightness array. The trigger timestamp is the only state the DMX thread keeps. This
  keeps the renderer jitter-proof and instantly responsive to live config changes, and
  matches the existing decay design.
- **Compositing with per-key beams.** Per-key beams (R33) and the active effect both
  write the same 40 channels. Default plan: combine **per channel by max** so a held
  key still reads through a sparse effect; revisit per effect if a full-field effect
  should instead take over. (Open decision below.)
- The render order in `dmx_thread._render()` is: zero frame → set per-beam mode on all
  active bars → write per-key decay → overlay active effects (the two existing
  `TODO(milestone-2)` lines).

### The first two effects

- **Laser lightning (R40).** While active, all 40 beams flash on/off at random, fast —
  a chaotic strobe/sparkle across the whole array. Re-randomised at a configurable
  flash rate (e.g. ~15–30 Hz) decoupled from the render tick, so the look is the same
  regardless of `tick_hz`. Random by nature, so this effect reads RNG each flash rather
  than being a pure function of elapsed time (the one exception to the closed-form rule).
- **Left-right wave (R41).** A bright "head" sweeps beam 0→39→0 (Larson/knight-rider);
  each beam, once the head passes, fades on a decay. The decay time is tuned to the
  sweep period so a beam is *almost* fully decayed by the time the head returns to it —
  a smooth trailing wave with no hard reset. Expressible closed-form: head position and
  each beam's "time since last passed" are both functions of elapsed time. Configurable
  sweep speed (full-sweep period) and decay time (or derive decay from the period).

### Resolved decisions

- **Chord definitions.** *Resolved 2026-07-20:* not fixed key-index sets at all — chord
  **quality** (major/minor), detected from held pitch classes (see Chord detection above).
  Supersedes the initial "reuse the QLC+ `ACCS` triples" plan.
- **Chord→effect mapping.** *Resolved 2026-07-20:* a config mapping `chord_effects` keyed
  by quality name, default **major → wave, minor → lightning**. Persisted in `config.json`;
  not yet in the web settings form (the numeric effect params are).
- **Web-UI exposure.** *Resolved:* the four numeric effect params (`lightning_flash_hz`,
  `lightning_on_fraction`, `wave_period_s`, `wave_decay_s`) are editable/persisted in the
  web UI; the quality→effect map is `config.json`-only for now.
- **Compositing.** *Resolved:* max-combine per channel, so a held per-key beam reads
  through a sparse effect (`_overlay_effects`). Revisit per effect if a full-field effect
  should instead take over.

## Open decisions (resolve during implementation)

- ArtNet library — must support sending DMX **and** ArtPoll/ArtPollReply discovery (or
  fall back to hand-rolling ArtPoll). Candidate: `stupidArtnet`; confirm it can receive
  ArtPollReply, otherwise add a small raw-UDP poller. TBD.
- Settings exposed in the web UI — at least: ArtNet target mode (broadcast/unicast),
  selected/entered unicast IP, universe, tick rate, fixture base addresses, master
  brightness, MIDI device selection, log level.
- Config file location and format (e.g. JSON/YAML next to the script).
- Whether to run Flask in a separate thread (simplest) or separate process — given the
  generous jitter budget, a thread is likely fine; revisit only if the web UI visibly
  disturbs the tick.

## Decided

- Language: **Python** — no rewrite in C/Go/Rust. Stick with the existing stack and do
  reasonably well on timing rather than over-optimizing.
- Target tick rate **~100 Hz**, user-configurable. Generous jitter budget (±50% is
  imperceptible here), so no hard-real-time work unless problems show up.
- Milestone 1 ships per-key beams only; effects come later (Milestone 2).
- Web stack: Flask.
- ArtNet target is selectable between broadcast and a specific unicast IP; the IP can be
  entered directly or picked from an ArtPoll discovery list. Universe is configurable.
